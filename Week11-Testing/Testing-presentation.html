A real story

A spreadsheet bug in
a famous economics paper

REINHART & ROGOFF · 2010

Two Harvard economists published a paper claiming countries with debt above 90% of GDP saw average real growth of −0.1%. The paper was widely cited in policy debates around austerity in the US and Europe.

Three years later, UMass economists examined the spreadsheet. An AVERAGE() formula didn't include all the rows it should have — five countries (Australia, Austria, Belgium, Canada, Denmark) were excluded. Once corrected, the figure changed from −0.1% to +2.2%.

No one had checked. There was no test. The spec — "compute the average growth across these countries" — had no automated guardrail.

Herndon, Ash & Pollin (2013), Does High Public Debt Consistently Stifle Economic Growth?

What testing is

Putting your code under attack —
on purpose, before anyone else does.

IS

  • trying to make the program fail
  • checking inputs you'd never type by accident
  • writing down what your code promises
  • a safety net for the version of you in 6 months

IS NOT

  • running your code once and seeing it work
  • exhaustive — you can't try every input
  • a substitute for thinking about the problem
  • something to add at the very end

Why this matters: almost every bug has a "smallest input that would have caught it" — and that input is rarely the one you'd type by accident. That gap is what testing is for.

Vocabulary

Types of tests you'll hear about

Unit test one function, in isolation "Does normalize_scores([1,2,3]) return [0, 0.5, 1]?"
Integration test several pieces, together "When my pipeline reads the CSV, cleans it, and saves output — does the output look right?"
Regression test "don't bring back old bugs" "Last month a date parsing bug bit us. We added a test. It must keep passing forever."
Smoke test "does it even run?" "Render the Quarto doc end-to-end. No errors? Good enough."

Today we focus on unit tests — they're the foundation. Once you can write good unit tests, the rest follow the same logic.

Why "just try it" doesn't work

A thought experiment

def divide(a, b): return a / b
01
Exhaustive testing is infeasible. Even simple functions have effectively infinite inputs.
02
Random inputs miss the interesting cases. 1,000 random numbers will almost never hit zero — but zero is exactly what breaks this.
03
"Typical" inputs miss everything. The input you'd type is the input you've already debugged for.

The key idea

Partitioning

Group inputs by the cases they exercise. Test one example from each group.

7 categories of input × 7 = 49 pairs to test. Not infinite. Not "a few." Just enough to cover every meaningfully different case.

Watch the boundaries — 0, empty inputs, type maxima, first & last elements. Bugs cluster at edges.

Two ways in

Blackbox vs Whitebox

BLACKBOX

Test against the spec.

You don't read the code.

  • catches misunderstandings of the spec
  • tests survive when implementation changes
  • may miss unusual code paths

WHITEBOX

Test against the code.

You read every branch.

  • catches edge cases hidden in if/else
  • aims for full coverage of paths
  • tests may need updating when code changes

In practice: do both. Blackbox first, whitebox second.

Spot the missing test

A colleague wrote mean_positive(x) — returns the mean of the positive values in a list. They tested it with these three inputs:

mean_positive([1, 2, 3])  →  passes ✓
mean_positive([5, 10, 15])  →  passes ✓
mean_positive([2.5, 3.5, 4])  →  passes ✓

They declare the function "tested." What inputs are they missing?

List at least three input cases that could expose a bug they haven't ruled out.

Pairs. 4 min to list, then we compare.
The point: three "happy path" tests look thorough but cover only one partition. Real testing means asking "what input would break this?" — not "what input do I expect to work?"

The workflow

The loop you'll actually use

STEP 1 Write a test based on the spec
→
STEP 2 Run it. Watch it fail. if it passes, is it really testing anything?
→
STEP 3 Fix the code smallest change to make it pass
→
STEP 4 Re-run the whole suite did your fix break something else?

The point of keeping tests around is not for now — it's for six months from now, when you change something and forget what depended on it.

This loop is what you'll do in Exercise 3 — write a failing test, fix the bug, re-run the suite.

Quality

What makes a good test

Clear

The name tells you what's being tested. If it fails six months from now, you can read it and understand what broke.

Fast

If running the suite is slow, you stop running it. Slow tests are tests that don't get used.

Isolated

One test per concept. Don't test five things in one block — when it fails, you won't know which one broke.

Deterministic

Same input → same result, every time. No random inputs, no dependency on the current date or wall-clock time.

A test you don't trust is worse than no test — it teaches you to ignore failures.
